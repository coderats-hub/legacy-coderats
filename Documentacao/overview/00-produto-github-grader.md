# Produto, GitHub e Microsoft Grader

## 1. Visão do produto

O CodeRats foi pensado como um produto acadêmico para apoiar organização de estudos, registro de progresso e colaboração entre estudantes de tecnologia.

Mais do que um aplicativo com telas e endpoints, ele funciona como uma resposta a um problema comum: transformar esforço de estudo em algo rastreável, compartilhável e avaliável ao longo do tempo.

Nesse cenário, GitHub e Microsoft Grader não são apenas ferramentas de apoio. Eles fazem parte do contexto em que o produto ganha sentido.

## 2. O que o produto faz

Na prática, o CodeRats permite:

1. criar e participar de grupos de estudo
2. registrar check-ins com evidências de atividade
3. acompanhar feed de progresso
4. curtir e comentar atividades
5. organizar perfis com vínculo ao GitHub
6. associar a evolução do aluno a sinais concretos de produção técnica

O produto não é só social. Ele também é um mecanismo de motivação, disciplina e rastreabilidade de entrega.

## 3. Papel do GitHub no produto

O GitHub é importante porque traz um sinal objetivo de atividade técnica.

No contexto do CodeRats, ele ajuda em quatro frentes:

* identidade técnica do usuário
* autenticação e vínculo de conta via OAuth
* leitura de commits e histórico de atividade
* suporte a feedback sobre produção de código

Isso é relevante porque reduz a dependência de percepção subjetiva. Em vez de medir só intenção de estudo, o produto consegue se aproximar de evidências reais de prática.

## 4. Papel do Microsoft Grader

O Microsoft Grader entra como parte do universo acadêmico de validação e avaliação.

Mesmo quando ele não é um componente interno do aplicativo, a existência de uma ferramenta de correção e aferição externa influencia a forma como o projeto é pensado:

* exige organização de entrega
* reforça disciplina de versionamento
* incentiva rastreabilidade do que foi feito
* ajuda a manter evidências do processo

Em ambientes acadêmicos, esse tipo de ferramenta aumenta a importância de manter repositório limpo, documentação coerente e histórico de mudanças bem registrado.

## 5. Por que GitHub e Grader importaram tanto

A importância dos dois está no mesmo ponto: eles tornam o processo verificável.

### GitHub

O GitHub foi importante porque:

* centralizou o código
* permitiu histórico de commits
* organizou branches e PRs
* facilitou revisão e colaboração
* serviu como base para integração com a aplicação

### Microsoft Grader

O Microsoft Grader foi importante porque:

* trouxe uma camada de avaliação formal
* aumentou a necessidade de consistência no material entregue
* reforçou a disciplina de documentação
* ajudou a transformar o projeto em algo avaliável por critérios objetivos

## 6. Impacto no desenho do projeto

Essas duas ferramentas impactaram diretamente a forma como o CodeRats foi organizado.

O projeto precisou ser construído pensando em:

1. rastreabilidade
2. reprodutibilidade
3. documentação consistente
4. versionamento claro
5. apresentação de resultados de forma objetiva

Isso explica por que o repositório não contém apenas código. Ele também contém documentação, artefatos, mídia e explicações de arquitetura.

## 7. Importância para o produto

Sem GitHub e Microsoft Grader, o projeto perderia parte importante do seu valor acadêmico e técnico.

O GitHub deu sustentação ao desenvolvimento real do software.

O Microsoft Grader deu contexto de avaliação e validou a necessidade de organizar o trabalho como produto entregável, não apenas como exercício de programação.

Em conjunto, eles ajudaram o CodeRats a sair do nível de ideia e entrar no nível de produto com processo, entrega e evidência.

## 8. Conclusão

O CodeRats foi fortemente moldado pelo ambiente acadêmico em que nasceu.

GitHub foi a base de colaboração, histórico e integração.

Microsoft Grader foi a referência de avaliação e formalização da entrega.

A importância disso foi alta porque obrigou o projeto a ser mais do que funcional: ele precisou ser explicável, versionado, documentado e defendível.
